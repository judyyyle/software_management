# 基础规则

## 一、 核心物理实体与资源网络 (Physical Entities & Resources)

现在，整个服务网络由四类实体构成：
1. **仓库 (Depot)**：位置固定，库存无限，是所有货物（含动态订单补货）的终极发货地，也是部分无人机的起降与维护中心，只有一个。
2. **卡车 (Truck/Mobile Base)**：
   - 在道路网络上移动，作为无人机的移动基站和移动微仓，只有一个。
   - 搭载 $$K$$ 架无人机，且卡车本身具备为无人机充电/换电的能力。
   - 给无人机提供起飞点与回收点。
3. **静态充换电站 (Static Stations)**：
   - **位置与分布**：在服务区域内离散分布的固定节点集合 $$S=\{s_1, s_2, \dots\}$$。
   - **补能模式**：统一采用**纯换电模式**。每个站点备用电池数量充足，无人机到站后无需等待电池资源，立即完成换电，耗时固定为 $$\tau_{swap}$$，完成后电量恢复为满电 $$E_{max}$$。卡车上的无人机补能同样采用换电逻辑，耗时为 $$\tau_{truck}$$。
   - **容量约束**：换电工位数量有限，同时服务的无人机数量上限为 $$Cap_s$$；若工位已满，后到无人机进入FIFO排队队列等待工位释放（电池充足不影响此排队，仅工位数量构成约束）。
4. **无人机 (Drones)**：
   - 具有最大载重量 $$C_{drone}$$和最大电池容量$$E_{max}$$。
   - 能量消耗随飞行距离和载重动态变化。

## 二、 订单模型设定 (Order Model)

订单属性保持不变，必须100%履约，即使迟到，也要送到：
- **订单集合**：静态预订单 $$O_{static}$$+ 实时动态订单$$O_{dynamic}$$。
- **订单向量**：$$Order_i=\langle Loc_i, q_i, t_{release}, t_{deadline} \rangle$$。软时间窗下，超时会产生惩罚成本。
- **货物同质性假设**：所有订单的货物视为**同质标准化货物**（如统一规格包裹），卡车携带的预留库存可直接满足任意动态订单需求，无需匹配特定货品。若未来扩展为异质货物场景，则到达前不可知的动态订单只能走模式C（仓-空直递）或触发卡车返仓取货，需单独建模处理。

## 三、 全局混合履约模式 (Expanded Fulfillment Mechanics)

引入充换电站后，无人机的作业半径被极大拓展。系统现在支持以下三种核心派送模式：
1. **模式 A：卡车直递 (Truck Direct)**
   - 卡车直接开到客户点派送大件或禁飞区订单，消耗卡车库存。
2. **模式 B：卡-空短途协同 (Truck-Drone Standard)**
   - 无人机携带卡车上的货物从卡车起飞，完成一单配送后，飞往卡车**后续路径中最近的充换电站或仓库节点**与卡车汇合，完成换电与回收上车。汇合点必须是充换电站或仓库，不得是客户点。
3. **模式 C：仓-空直递 (Depot-Drone Direct)**
   - 针对距仓库较近的动态订单，或者卡车上没有对应货品的订单，无人机直接从仓库带货飞抵客户，然后返回仓库或最近卡车。

# 问题构建
---

## 一、 优化目标 (我们要把什么降到最低？)
**时间成本与惩罚**：
   - 所有订单履约完成时刻之和（$$\sum_i C_i$$，加权完成时间），反映整体服务效率。
   - 超时惩罚：对于带有软时间窗的订单，如果送达时间晚于客户的期望时间，按超出时长计算惩罚金额。

---

## 二、 核心决策 (算法需要输出什么结果？)

1. **卡车的路线规划**：卡车从仓库出发，具体要经过哪些客户点（直递模式），要经停哪些充换电站（作为微仓/回收点），以及先后顺序是什么。
2. **无人机的任务分配与模式选择**：每个订单交给谁送？是卡车自己送（模式A），还是无人机从卡车起飞送（模式B），还是无人机直接从仓库送（模式C）。**每架无人机每次飞行仅携带并配送一个订单。**
3. **无人机的落脚点决策**：无人机完成配送后，须从以下三种选项中选择下一个落脚位置：
   - 选项①：飞往某一充换电站换电，等待卡车接载继续执行任务；
   - 选项②：当电量不足以直接完成下一段路径时，飞往最近充换电站补能，再转飞目标节点；
   - 选项③：直接飞回仓库。
   落脚点的选择受当前剩余电量、各节点距离、卡车预计到达时间共同约束。

## 三、 约束条件梳理 (必须遵守的物理与业务规则)

我们把约束分为四大板块来梳理：

### 1. 基础网络与履约约束 (最基本的规矩)
- **闭环规则**：卡车必须从仓库出发，最后回到仓库。所有的无人机也必须从仓库出发（无论是自己飞出去还是坐在卡车上），最后全部回到仓库。**无人机在完成最终任务后飞回仓库的路段，同样须通过前瞻能量校验**；若当前电量不足以直飞回仓库，须在途中先停靠充换电站补能，不得强行返仓。
- **必达规则**：无论是静态订单还是已经触发的动态订单，每一个客户都必须且只能被服务一次（不能漏送，也不能两架无人机同时去送同一单）。每架无人机每次飞行仅携带并配送一个订单。
- **物理绑定规则**：无人机只有在和卡车处于同一个物理位置（仓库或充换电站）时，才能坐上卡车。不能在半路凭空上车。

### 2. 时空协同约束 (卡车与无人机的“握手协议”)
时空协同约束本质是对卡车与无人机在特定节点的**时间窗匹配**与**物理位置重合**的双重约束，保证两者能在正确的时间、正确的地点完成交接。
- **起飞同步**：无人机从卡车起飞（模式B）须满足：① 卡车已到达该节点（$$t_{truck,arrive} \leq t_{drone,takeoff}$$）；② 卡车上有对应订单货物；③ 卡车须等待无人机完成起飞准备后方可离开（$$t_{truck,depart} \geq t_{drone,takeoff}$$）。
- **回收同步（关键）**：无人机飞往充换电站等待卡车接载，须满足：无人机到站时刻 $$t_{drone,arrive,s} \leq t_{truck,depart,s}$$（卡车离开该站的时刻）。若卡车离站后无人机才到达，不再记为硬失败，而是自动转入“最近可达站点/仓库等待”，并施加较大惩罚，用于遗传算法中的软约束处理。
- **等待方向性**：允许无人机先于卡车抵达充换电站排队等待（无人机等卡车）；卡车在充换电站等待无人机须设置最大等待时间上限 $$T_{wait,max}$$，统一取值为 10s。超时则卡车继续行驶，被等待的无人机自动转入“最近可达站点/仓库等待”，并给予较大惩罚。
- **节点停留**：卡车或无人机到达客户节点须停留卸货交接时间 $$\delta_{service}$$；无人机到达充换电站须停留换电时间 $$\tau_{swap}$$（换电模式）；无人机搭乘卡车行驶期间视为休整待命状态，不消耗电量。
- **等待期能耗**：无人机在充换电站等待工位释放期间，或换电完成后等待卡车到来期间，均视为**降落停机状态**（站内配备停机坪），旋翼关闭，能耗近似为 $$0$$。此规定直接影响前瞻能量校验的边界条件：无人机降落后的等待时长不消耗电量，仅飞行阶段计入电量损耗。
- **空间唯一性**：**卡车与无人机的上下车交接只能发生在仓库或充换电站**，不允许在道路中间拦截行驶中的卡车，也不允许在客户点完成上下车交接。因此模式B中，无人机完成配送后的汇合回收点必须是充换电站或仓库，而非客户点（详见基础规则模式B定义）。



### 3. 前瞻能量与电池约束 (绝不坠机的红线)
- **能量守恒**：无人机每次飞行的耗电量必须从电池中扣除。在充换电站或坐上卡车后，电量恢复为满电。
- **绝对安全红线**：在任何时刻，无人机的电量都不能低于安全底线。
- **前瞻能量校验**：算法在指派无人机接任何一单之前，必须进行一次“模拟推演”——确保无人机当前的电量，足够它飞到客户那里完成卸货，并且还能继续飞到一个合法的落脚点（某个充换电站、或者仓库），且降落时电量依然高于安全红线。如果推演失败，这单就不允许这么派。
- **携货禁停规则**：无人机在**携带货物飞行期间**（从起点出发至完成卸货前），不允许中途降落于任何充换电站进行补能。因此模式C（仓-空直递）中，无人机必须在从仓库出发时确保电量足以支撑“仓库 → 客户 → 落脚点”的完整三段路径，前瞻校验须一次性评估全程能量充足性。

### 4. 载重、库存与容量约束 (装载的物理极限)
- **无人机载重**：无人机单次携带的货物重量，不能超过其最大载荷 $$C_{drone}$$。
- **充换电站补能约束**：采用**纯换电模式**，每个站点备用电池数量充足。无人机到站后立即开始换电，耗时固定为 $$\tau_{swap}$$，完成后电量恢复为满电 $$E_{max}$$。不存在因电池不足而等待的情况。
- **充换电站工位排队约束**：每个充换电站同时可服务工位数上限为 $$Cap_s$$。若无人机到站时工位已满，进入FIFO队列等待工位释放；等待时长须纳入无人机时间轴规划，影响其后续任务的最早可用开始时间。

## 四、 相关代码文件（参数与约束来源）

为了便于后续改参和扩展，下面把当前贪心基线中与本文规则直接对应的代码文件集中列出。这里仅记录职责与来源，不固化具体数值。

- [backend/solver/greedy_baseline.py](../backend/solver/greedy_baseline.py)：极简贪心基线入口，默认采用模式 B_WAIT，载重超限时回退模式 A。
- [backend/solver/greedy_mmce.py](../backend/solver/greedy_mmce.py)：多模式贪心主实现，负责模式 A / B / C / B_WAIT 的候选评估、前瞻能量校验、回收点选择、卡车路径构建与评分。
- [backend/solver/decision_engine.py](../backend/solver/decision_engine.py)：调度编排层，负责调用求解器并把分配结果落到订单与实体状态上。
- [backend/config/loader.py](../backend/config/loader.py)：统一加载求解器时长参数、无人机参数与评分参数，是运行时参数的主要入口。
- [backend/config/drone_params.yaml](../backend/config/drone_params.yaml)：无人机机型物理参数配置，包含气动参数、巡航速度、载重能力、电池容量与安全余量。
- [backend/core/entities/drone.py](../backend/core/entities/drone.py)：无人机物理实体定义，包含机型、载荷、电池、飞行状态机、航路执行与能耗计算。
- [backend/core/entities/truck.py](../backend/core/entities/truck.py)：卡车实体定义，包含行驶速度、库存、无人机停靠与换电/回收相关能力。
- [backend/core/entities/primitives.py](../backend/core/entities/primitives.py)：共享原语与状态机定义，包含位置、来源类型、航路点动作、无人机状态与卡车状态。
- [backend/core/entities/swap_station.py](../backend/core/entities/swap_station.py)：静态充换电站实体定义，负责站点容量、换电工位与排队语义。
- [backend/core/entities/charging_host.py](../backend/core/entities/charging_host.py)：充换电宿主抽象基类，承载卡车与站点的通用停靠/换电接口。
- [backend/environment/state/entity_manager.py](../backend/environment/state/entity_manager.py)：实体装载与运行时参数注入入口，负责把配置中的无人机与调度参数同步到仿真实体。

如果后续需要继续拆分约束，优先从上述文件扩展，而不是在问题描述正文里继续绑定固定数值。
