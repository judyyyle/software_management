你的想法整体是对的，而且我觉得这是当前机制里比较应该补的一块：现在 planner_invalidated_for_truck_order 对 arrived_before_eta_ref 和 eta_late_exceeds_threshold 太保守了。

当前代码里三种原因都会被一刀切处理：

node_not_in_truck_plan
arrived_before_eta_ref
eta_late_exceeds_threshold
只要是 INVALIDATED，env 就释放 reservation，并且如果无人机正在 RETURN_TO_RENDEZVOUS 或 WAITING_FOR_TRUCK，直接 fallback。

这个处理对 node_not_in_truck_plan 合理：新卡车计划里已经没有这个站点了，继续等没有意义。

但对后两种情况，确实有“重新计算后仍可执行”的空间。

对 arrived_before_eta_ref
这表示新卡车 ETA 比原 ETA 更早。当前 planner 把它标 invalidated，是因为它违反了“不能早于原 ETA”的 reservation 约束。

但执行层上要分情况：

如果无人机已经到达汇合点，也就是 WAITING_FOR_TRUCK，卡车更早通常不是坏事。只要卡车还没错过当前时刻，或者这个 outcome 是当前重排计划里的未来到达，就应该允许继续。
如果无人机正在飞向汇合点，也就是 RETURN_TO_RENDEZVOUS，需要重新估计无人机到站时间。如果：
uav_arrival + rendezvous_execution_margin_sec <= new_truck_eta
那仍然可执行，不应该直接 fallback。

如果新卡车 ETA 已经早到到无人机赶不上，那 fallback 才合理。

对 eta_late_exceeds_threshold
这表示新卡车 ETA 比原 ETA 晚，并且超过了 reservation drift 阈值。当前配置里这个 drift 阈值是 60s，但真正执行层允许的 rendezvous 等待上限是 rendezvous_max_wait_sec: 240。

所以现在会出现一种偏保守情况：

新 ETA 晚了 80s
> drift 阈值 60s，所以 planner invalidated
但无人机到站后只需要等 80s
< rendezvous_max_wait_sec 240s，其实还能等
你的建议正好能修这个问题。判断应该基于：

对正在飞的无人机：重新估计 UAV 到达汇合点时间，然后算 new_truck_eta - uav_arrival 是否在允许范围内。
对已经到达的无人机：用 wait_started_at 算到新卡车 ETA 的总等待时间是否超过 rendezvous_max_wait_sec。
也就是类似：

arrived_before_eta_ref:
  需要确认 UAV 仍能早于新 truck ETA 到达，或已经在等。

eta_late_exceeds_threshold:
  需要确认 UAV 到达后等待到 new truck ETA 不超过 rendezvous_max_wait_sec。
我建议的实现口径
我会把 planner 的 INVALIDATED 分成两层理解：

planner 层：相对旧 reservation 契约失效。
env 执行层：重新评估后，是否仍可安全保留。
也就是说，保留 PlannerBridge 的诊断结果，但在 env 的 _apply_planner_reservation_outcomes() 里新增一个执行层复核：

node_not_in_truck_plan:
  保持现状，释放 reservation / fallback。

arrived_before_eta_ref:
  如果 new_eta 仍满足当前 UAV 可赶上，则保留 reservation，并更新 commit.planned_truck_arrival_time = new_eta。
  否则释放 / fallback。

eta_late_exceeds_threshold:
  如果按 new_eta 计算等待不超过 rendezvous_max_wait_sec，则保留 reservation，并更新 ETA。
  否则释放 / fallback。
这里还要注意状态差异：

RETURN_TO_RENDEZVOUS：用当前无人机位置重新估算到 recover node 的飞行时间。
WAITING_FOR_TRUCK：不需要再算飞行时间，用当前等待开始时间或当前时刻算等待是否超限。
DELIVERY_SERVICE / DELIVERED：也可以复核，但当前代码 invalidated 后不会立刻 fallback，后面送达 service 结束会重新选点。第一版可以先不动这部分，避免扩大影响。
关于记录卡车每次重规划
这个也非常有必要。现在 episode 级日志只有：

fallback_cause_counts.planner_invalidated_for_truck_order

但无法知道 invalidated 的子原因：

我的评价
你的方案方向很好，但我会稍微改成更明确的原则：

node_not_in_truck_plan：继续立即释放 reservation。
arrived_before_eta_ref：不要直接失败，按当前 UAV 是否还能赶上新 ETA 复核。
eta_late_exceeds_threshold：不要用 60s drift 阈值直接失败，改用执行层等待阈值复核。
记录 planner replan events，否则后面无法判断是 planner 策略问题、ETA 阈值问题，还是卡车路线本身问题。
这会更符合实际执行语义，也能减少当前日志里大量 planner_invalidated_for_truck_order 带来的 Mode C 失败。
