# PPO 在线调度模式与运行时路径方案

## 目标

我们需要保留当前 PPO 训练模式，同时新增一个在线调度模式。

训练模式继续服务离线训练、验证、回放统计。  
在线调度模式服务前端实时运行、实时决策展示、路径显示、动态订单接入、以及后端位置真值检查。

这两个模式不能各自长出一套不同的调度语义。  
我们需要的是：

- 一套共享的环境真值
- 两套不同的运行驱动方式

环境真值负责订单、车辆、无人机、事件、能耗、回收、fallback、reservation、送达、充电这些业务语义。  
运行驱动方式只决定它是给训练器批量滚动，还是给前端按时间窗实时播放。

## 我们需要做什么

我们需要把当前 `TrainingEnvAdapter` 从“只能 `step()` 跳到下一个决策点的训练环境”改成“既能离线滚动，也能在线播放的统一环境核心”。

我们需要保留当前 `step(action)` 接口，继续给训练和验证使用。  
同时我们需要新增一组面向在线模式的公共接口：

- `peek_next_event_time()`：查看下一个事件时刻
- `peek_next_decision_time()`：查看下一个决策时刻
- `advance_to_time(t_target)`：只推进到指定仿真时间，不跨越不该跨越的决策边界
- `apply_decision(action)`：在当前决策点应用动作，但不隐式替代整个在线播放流程
- `build_runtime_snapshot()`：返回已经和当前 `t_now` 对齐的位置、状态、路径、决策信息

当前 `step(action)` 可以保留为离线包装器：

1. 应用动作
2. 自动推进
3. 一直走到下一个决策点或结束

这样训练模式不需要重写。  
在线模式则不再直接依赖现在这种“一步跳几十秒”的 `step()` 驱动前端。

这里要补一条第一阶段就冻结的接口边界：

- 在线模式下，`WAIT` 的等待时长、唤醒条件、下一次决策暴露时机，全部复用当前训练侧 `TrainingEnvAdapter` 既有实现
- 在线模式不额外发明第二套 `WAIT` 规则
- `advance_to_time(t_target)` 只负责按既有环境真值推进，不负责重新定义 `WAIT` 应该等多久

也就是说，在线模式展示的是训练侧真值运行过程，而不是再造一套“看起来像训练结果”的近似播放器。

## 执行阶段划分

基于当前代码基础和上线风险，执行按四个阶段推进。

### 第一阶段：冻结 MVP 语义与公共接口边界

目标：

- 把在线模式和训练模式必须共享的语义先写死
- 明确第一版在线运行真正需要暴露的最小接口
- 删掉第一版不需要的强绑定和历史追踪要求，避免实现时边写边猜

本阶段产出：

- 冻结 `WAIT` 语义：完全复用训练侧实现
- 冻结在线结束条件：订单结束后，继续推进，直到所有实体都回到仓库且没有在途执行链路
- 冻结决策顺序：同一 `sim_time` 下完全复用 `TrainingEnvAdapter` 当前决策队列顺序
- 冻结路径协议 MVP：前端只消费当前 active path snapshot，不要求路径和决策事件做一一绑定
- 冻结在线公共接口最小集合：`peek_next_event_time()`、`peek_next_decision_time()`、`advance_to_time()`、`apply_decision()`、`build_runtime_snapshot()`

### 第二阶段：拆出统一环境核心

目标：

- 在不破坏训练链路的前提下，把 `TrainingEnvAdapter` 从“只会 `step()` 的训练壳”拆成“离线/在线共用的环境核心”

本阶段产出：

- 实现在线公共接口
- 保留 `step(action)` 作为离线包装器
- 让训练、验证继续走原接口，在线运行改走新接口

### 第三阶段：上线在线运行驱动与前端真值展示

目标：

- 新增在线运行驱动器
- 让前端真正按时间窗看到环境真值、当前有效路径和决策状态

本阶段产出：

- `OnlinePolicyRuntimePlayer`
- `DECISION_PENDING` / `DECISION_APPLIED` / `EXECUTION_HARD_FAILED` 事件流
- `FULL_SNAPSHOT` / `TICK` 基于统一环境真值输出
- 前端按实体全量替换当前路径层，不做 append

### 第四阶段：补强路径账本与未来避障

目标：

- 在在线 MVP 跑通后，再补历史追踪、路径来源绑定、未来避障和训练/在线统一运动代价口径

本阶段产出：

- 更完整的 path ledger 生命周期
- 决策事件与路径来源绑定
- `estimate_motion_metrics(...)`
- `resolve_motion_path(...)`
- 训练/在线统一避障代价与执行语义

## 在线调度模式要怎么运行

我们需要新增一个在线运行驱动器，例如 `OnlinePolicyRuntimePlayer`。

它的职责不是自己维护第二套仿真语义。  
它只做四件事：

- 按 wall-clock 定时唤醒
- 计算本轮应该推进到哪个仿真时刻
- 让环境只推进到这个时刻或推进到决策边界
- 在出现决策点时触发一次真实模型推理，并把决策事件发给前端

在线模式下，前后端看到的行为应该是：

- 仿真时间按设定倍率向前走
- 动态订单只在 `create_time <= sim_time` 时出现
- 如果当前没有决策点，环境继续按时间窗推进
- 如果当前碰到决策点，播放器停在该决策点
- 模型开始真实推理
- 前端收到“决策待处理”状态
- 推理结束后，前端收到“决策已应用”事件，并在终端进行打印
- 环境再继续推进

这样模型不是“上帝视角直接拿走所有未来订单”。  
模型只能看到当前仿真时刻已经出现的订单，也只能在当前决策点上做决策。

## 我们需要如何处理推理耗时

在线模式必须把“真实推理耗时”从系统里漏出来。

我们至少需要暴露：

- 决策开始时间
- 决策完成时间
- 推理耗时毫秒数
- 当前决策对应的无人机
- 当前决策触发原因
- 应用的动作内容

前端应该能看到：

- `DECISION_PENDING`
- `DECISION_APPLIED`
- `EXECUTION_HARD_FAILED`

第一版在线模式里，建议在决策点暂停仿真推进，等推理完成后再继续。  
这能保证前端时间轴、订单可见性、位置更新、路径替换都最容易解释。

这里要明确区分“决策提交阶段”和“执行结果阶段”：

- `DECISION_PENDING` 表示当前决策点已经暴露，运行时正在等待模型完成本次推理
- `DECISION_APPLIED` 表示该决策已经成功提交到环境，并开始进入后续执行
- `EXECUTION_HARD_FAILED` 表示该决策已经提交，但其执行结果最终导致 actor drone 进入 hard failure state

也就是说，这里的 hard failure 不是“决策没有提交成功”，而是“决策已应用，随后在执行过程中触发了硬故障后果”。

如果后续确实需要把“算法计算本身也算作仿真中的时间消耗”，那就单独引入一个显式参数，例如：

- `decision_service_time_sec`

它表示“决策结果从开始计算到真正生效，需要消耗多少仿真时间”。  
这个时间要明确建模，不能靠现在这种后台线程隐式快进来假装。

## 在线模式不能直接沿用当前 is_done 语义

当前训练环境的 `is_done()` 适合离线 episode。  
它会在这些情况下结束：

- 到达 horizon
- 没有 pending、没有 assigned、没有背景订单
- 所有无人机都失败

这套语义不适合在线调度模式。

在线调度模式里，我们需要单独定义运行结束策略。  
默认结束条件要明确成“订单结束 + 全设备回仓”双条件同时成立：

- 订单侧结束：没有 `pending_orders`，没有 `assigned_orders`，也没有未来尚未出现的动态订单
- 设备侧结束：所有卡车和无人机都已经回到仓库，且没有任何实体处于在途或执行中状态

只有上面两组条件同时满足时，在线调度运行才结束。  
仅仅“当前订单池空了”还不够，因为这时可能仍然存在：

- 卡车正在返仓途中
- 无人机正在返航、等待回收、充电收尾
- 已提交决策对应的执行链路尚未完全结束

因此在线结束判定至少还要额外检查：

- 当前没有 active truck path
- 当前没有 active drone flight leg
- 当前没有待处理决策事件

也就是说，在线模式不能因为订单刚好送完就立刻退出。  
它必须继续推进，直到订单生命周期和载具收尾生命周期都真正闭合。

如果业务上仍然需要人工中止或防止无限等待，可以再额外保留两个保护条件：

- 显式停止
- 到达最大仿真时间

## 初始卡车路径规划要纳入在线模式的标准入口

在线模式启动前，我们需要先把初始卡车路径规划好，并且把这条路径作为运行时真值的一部分固定下来。

当前已经有一条可直接复用的链路：

- Phase 4 路线产物已经输出 `stops + segments + geometry`
- PPO 运行时已经按 `planned_segments` 沿真实路网 geometry 推进卡车位置
- `poisson` 巡站补路也已经改成补全真实路网 segment，而不是 stop-to-stop 直线拼接

在线模式启动时必须直接复用这条现有链路，不能再单独做一套“在线专用卡车路线重建”。

在线模式后续如果要重规划卡车路线，也必须继续走同样的数据结构：

- 保留完整路线几何
- 保留新的停靠点时间窗
- 保留旧路径失效和新路径生效的边界

## 路径显示不能靠前端猜

我们需要把“决策到路径显示”的转换放在后端。

前端只负责显示：

- 当前真值位置
- 当前有效路径
- 决策事件
- 路径失效和替换

前端不应该自己根据订单、车站、载具状态去猜一条路径。  
否则前端会复制一套后端逻辑，最后既查不出真值问题，也容易和执行层不一致。

## 在线模式要在现有路径账本基础上补齐语义

当前后端已经能输出运行时 `paths` 快照：

- 卡车输出当前 active segment 的剩余路网路径
- 无人机输出当前 active flight leg 的剩余路径
- 前端 telemetry 已经能收到 `paths.trucks` 和 `paths.drones`

但这还只是“当前有效路径快照”，还不是完整的在线调度路径语义。  
第一版在线模式里，路径协议先收敛到当前显示所必需的最小集合：

- 更明确的 `status` 生命周期
- 路径切换原因，例如新决策改写、执行完成、异常中断

第一版不要求路径账本回答“这条路径由哪次决策生成”。  
也就是说：

- 第一版不强制要求 `source_decision_id`
- 第一版不强制要求 `source_plan_version`
- 第一版先把“当前有效路径能稳定显示和替换”做好

如果后续要做回放、追责、路径来源排障，再在第四阶段补事件与路径来源绑定。

这里的 `motion_mode` 仍然要保持显式字段，不能把协议写死成“卡车一定走路网、无人机一定直线”。  
后续至少要继续兼容：

- `road_network`
- `straight_line`
- `obstacle_avoiding`
- `riding_with_truck`
- `stationary`

## 过期路径必须被替换，不能一直挂在地图上

路径显示不是“历史回放线”。  
它表示的是“当前执行中的有效路径”。

所以我们需要明确这些规则：

- 新决策生效后，旧路径被标记为 `superseded`
- 当前路径执行结束后，被标记为 `completed`
- 前端默认只显示 `active` 的路径
- 如果需要回看历史，再单独开历史模式

这正好满足你要的“路径一旦被新决策改写，旧路径就不能继续显示”。

结合当前代码，这里还要补一条实现约束：

- 前端对 `paths` 的处理应该是“按实体全量替换当前路径层”，不能做 append 累积

因为当前后端发的是 active path snapshot，不是历史 path stream。

## 跟随卡车的无人机不单独显示路径

当无人机处于这些状态时，不应该再独立画一条飞行路径：

- `riding_with_truck`
- `charging_on_truck`
- 其他明确跟随卡车的位置绑定状态

这些状态下，无人机位置直接从属卡车。  
前端只显示卡车路径，不重复显示无人机路径。

如果无人机刚从卡车起飞，路径从“机随车”切换到“独立飞行”这一刻开始，才生成新的无人机有效路径段。

## 在线模式必须外露决策事件

我们需要给前端一个显式的决策事件流，而不是只在 `stats` 里塞一个当前决策摘要。

每条决策事件至少要有：

- `decision_id`
- `sim_time`
- `wall_time_ms`
- `event_seq`
- `drone_id`
- `trigger_type`
- `trigger_station_id`
- `candidate_summary`
- `selected_action`
- `selected_order_id`
- `selected_mode`
- `selected_recover_node`
- `inference_latency_ms`
- `status`
- `actor_drone_final_state`
- `failure_type`

其中字段语义要固定：

- `decision_id` 表示这次决策点自身的业务主键
- `event_seq` 表示事件流内严格单调递增的序号，供前端重连恢复和去重
- `status` 至少要支持 `DECISION_PENDING`、`DECISION_APPLIED`、`EXECUTION_HARD_FAILED`
- `actor_drone_final_state` 用于显式外露该事件落地后的 actor drone 状态
- `failure_type` 仅在失败结果事件中必填，例如 `airborne_energy_failure`

也就是说，`EXECUTION_HARD_FAILED` 不是一条孤立的“无人机坏了”事件。  
第一版里它只需要作为执行结果事件被外露出来。  
如果后续要把它严格回指到某条历史 `DECISION_APPLIED` 事件，再在第四阶段补充来源绑定字段。

这里还要冻结一条顺序约束：

- 同一 `sim_time` 下多无人机同时触发时，在线模式完全复用 `TrainingEnvAdapter` 当前决策队列的入队顺序和消费顺序
- 在线运行驱动器不额外引入第二套重排序逻辑
- `event_seq` 只是在这个既有顺序上做严格单调递增编号，不参与重新排序

在数据面上，我们还要固定一条实现原则：

- 后端全量维护一份 append-only 的 `decision event ledger`
- 每条事件写入 ledger 时分配严格单调递增的 `event_seq`
- WebSocket `FULL_SNAPSHOT` 和 `/state` 默认只携带最近窗口，例如最近 `N` 条 `recent_decision_events`
- 快照中同时返回 `latest_event_seq`
- 完整历史不通过实时快照全量回灌，而是通过单独事件接口按序拉取，例如 `/policy/events?after_seq=...`

这样做的目的很明确：

- 后端保留完整历史，便于追责、回放、排障
- 前端重连时先用快照恢复当前真值，再用最近窗口补齐邻近事件
- 长时间运行时，快照大小不会随着历史无限增长

如果前端离线过久，导致缺失区间已经落在最近窗口之外，则恢复策略要明确成：

- 以前端收到的最新 `FULL_SNAPSHOT` 为当前真值基准
- 不保证自动补全窗口之外的全部历史事件
- 若前端确实需要完整补历史，则再显式调用事件历史接口分页拉取

这样后端账本和前端展示链路在后续增强阶段才能做三件事：

- 展示 PPO 此刻为什么在决策
- 在需要时把当前路径和某次决策对应起来
- 检查是否存在漏决策、重复决策、错误决策

## 动态订单的可见性规则要固定

在线模式里，订单是否对模型可见，只由仿真时间决定：

- `order.create_time <= sim_time`，订单可见
- `order.create_time > sim_time`，订单不可见

前端到没到这一时刻、模型计算快不快、wall-clock 过了多久，都不应改变这条规则。

这样动态订单的出现、模型看到订单的时刻、前端看到订单的时刻，都会统一到同一个 `sim_time` 上。

## 未来避障要怎么接入

未来避障不能只做在前端，也不能只做在最终飞行显示层。

我们需要把“路径估计”和“路径执行”拆开，但接口要统一。

我们需要引入两个后端能力：

- `estimate_motion_metrics(...)`
- `resolve_motion_path(...)`

`estimate_motion_metrics(...)` 用于调度筛选和重校验。  
它返回：

- 路径长度
- 预计飞行时间
- 预计能耗
- 是否可达

`resolve_motion_path(...)` 用于真正生成执行路径。  
它返回：

- 完整路径点序列
- 路径长度
- 飞行时间
- 能耗
- 采用的运动模式

这样就能把未来避障接成统一框架。

## 避障不应该在所有候选上暴力全量求精

调度阶段如果对每一个订单、每一个回收点、每一个模式都去跑一次完整避障规划，代价会很高。

更稳的做法是两层筛选：

- 第一层用便宜的估计器做粗筛
- 第二层只对少量候选做精确路径求解

具体规则应该是：

- 粗筛阶段可以使用直线距离、带保守系数的绕飞长度、或者轻量级长度估计函数
- 对进入 shortlist 的候选，再调用精确路径长度函数
- 对最终选中的动作，在提交前再做一次精确校验
- 在真正开始执行飞行段时，生成实际执行 polyline

这样能同时满足速度和一致性。

## 训练模式和在线模式都要用同一套避障口径

未来引入避障后，训练模式和在线模式不能一个用直线、一个用避障真路径。

否则会出现：

- 训练时学的是一套代价
- 在线跑的是另一套代价
- 策略分数、可行性判断、返航判断、reservation 判断全部漂移

正确做法是：

- 训练和在线都调用同一套 `estimate_motion_metrics(...)`
- 训练和在线都按同一口径算时间、能耗、可达性
- 在线模式额外调用 `resolve_motion_path(...)` 生成显示和执行路径

如果训练期嫌精确避障太慢，可以保留“粗筛 + 精筛”的两层结构，但规则必须一致，不能临时拍脑袋换一套。

## 无人机飞行账本还需要继续补强

当前 `FlightLeg` 已经不再只是“起点 + 终点 + 起飞到达时间”。

现在已经有：

- `motion_mode`
- `path_points`
- `route_version`
- `energy_cost_j`

但它的执行语义仍然是直线飞行账本。  
为了接入未来避障和在线决策事件绑定，仍然需要继续补：

- `path_length_m`
- `speed_profile`
- `source_decision_id`
- 非直线 polyline 的逐段执行语义

这样无人机飞行既能保留当前直线版，也能无缝扩到未来避障版。

## 我们需要保留手动调度和 PPO 在线模式的边界

当前 classic `/dispatch` 和 PPO 在线模式不应该混成一条入口。

我们需要明确：

- classic `/dispatch` 继续服务人工触发的基线调度
- PPO 在线模式使用单独的 runtime 入口
- 两者都可以复用同一张地图和同一套前端路径显示协议

这样前端展示层统一，后端运行模式分明，不会互相污染。

## 最终落地原则

我们需要做的不是再造一套前端补间动画。  
我们需要做的是把 PPO 在线运行变成一个有真值路径、有真值时间、有真值决策事件的运行系统。

核心原则只有几条：

- 共享一套环境真值，不共享两套语义
- 在线模式按时间窗推进，不再用离线 `step()` 直接驱动前端
- 决策事件必须外露
- 动态订单只按 `sim_time` 可见
- 卡车必须沿真实路网 geometry 运动
- 无人机路径协议不能写死成直线，要预留避障模式
- 路径显示由后端给真值，前端只负责显示
- 旧路径被新决策改写后必须失效
- 跟车无人机不重复显示独立路径
- 训练和在线最终要共用同一套路径代价和可达性口径
